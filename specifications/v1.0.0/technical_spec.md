# v1.0.0 (MVP): 통합 기술 명세서 (Technical Specification)

이 문서는 AI 페이스메이커 v1.0.0 개발을 위한 아키텍처, 기능별 구현 요건 등 모든 기술적 내용을 통합하여 정의합니다.

## 1. 기능별 상세 명세

### **Feature 0: 사용자 인증 (User Authentication)**

#### 1. 사용자 경험 목표 (UX Goal)

- **User's Job (JTBD):** When AI 페이스메이커에 자신의 아이디어와 계획을 기록하고 싶을 때, **김성장**은 그 내용이 지속적으로 관리될 수 있는 **자신만의 전용 작업 공간을 만들고 싶어한다.**
- **Success Metric:** 이 기능의 성공은 신규 가입 유저의 '가입 완료 후 첫 기능 사용률'이 높고, 기존 유저의 '로그인 실패율'이 낮게 유지되는 것으로 판단할 수 있다.

#### 2. 구현 요건 (Acceptance Criteria)

- **(정상 시나리오 - 회원가입)** **Given** 사용자가 회원가입 페이지에서 유효한 이메일과 비밀번호 정책을 만족하는 정보를 입력하고, **When** 가입 요청 후 자신의 이메일 받은 편지함에서 인증 링크를 클릭하면, **Then** 계정 활성화가 완료되고, 사용자는 로그인하여 서비스의 메인 화면으로 이동할 수 있다.
- **(정상 시나리오 - 로그인)** **Given** 가입된 사용자가 로그인 페이지에서, **When** 자신의 이메일과 비밀번호를 정확하게 입력하면, **Then** 성공적으로 인증되어 자신의 작업 공간으로 이동한다.
- **(예외 케이스 - 이메일 미인증)** **Given** 사용자가 이메일 인증을 완료하지 않은 상태에서, **When** 로그인을 시도하면, **Then** "이메일 인증을 먼저 완료해주세요." 라는 메시지를 표시하고, 인증 메일 재발송 옵션을 제공한다.
- **(예외 케이스 - 비밀번호 정책 위반)** **Given** 사용자가 회원가입 시, **When** 비밀번호 정책에 맞지 않는 비밀번호를 입력하면, **Then** "비밀번호는 최소 10자 이상이어야 하며, 영문 대/소문자, 숫자, 특수문자를 모두 포함해야 합니다."와 같이 명확한 에러 메시지를 즉시 표시한다.
- **(예외 케이스 - 로그인 실패)** **Given** 사용자가 로그인 시, **When** 잘못된 이메일 또는 비밀번호를 입력하면, **Then** "이메일 또는 비밀번호가 올바르지 않습니다."와 같은 통합된 에러 메시지를 표시한다.

### **Feature 1: AI 기반 백로그 정의**

#### 1. 사용자 경험 목표 (UX Goal)

- **User's Job (JTBD):** When 머릿속에 '...을 해야 하는데' 라는 막연한 아이디어가 떠올랐을 때, **김성장**은 그것이 다른 생각들(소음)에 묻히기 전에, **명확하고 실행 가능한 일(신호)로 붙잡아 두고 싶어한다.**
- **Success Metric:** 이 기능의 성공은 'AI로 구체화하기' 버튼을 누른 사용자 중, 최종적으로 백로그를 등록 완료하는 비율(전환율)이 높게 유지되는 것으로 판단할 수 있다.

#### 2. 구현 요건 (Acceptance Criteria)

- **(정상 시나리오)** **Given** 사용자가 백로그 목록에 제목만 입력한 상태에서, **When** [AI로 구체화하기] 버튼을 누르고 AI의 질문에 답변하면, **Then** 대화 내용을 바탕으로 '제목', '설명', '완료 조건'이 자동으로 채워진 백로그가 표시되고, 이를 등록할 수 있다.
- **(정상 시나리오 - 스트리밍 응답)** **Given** 사용자가 AI와 대화 중일 때, **When** AI가 답변을 생성하면, **Then** 사용자는 마치 실제 채팅처럼 텍스트가 실시간으로 타이핑되는 스트리밍(Streaming) 응답을 볼 수 있다.
- **(예외 케이스 - 네트워크/서버 오류)** **Given** 사용자가 AI와 대화 중, **When** 서버 측 문제나 네트워크 문제로 응답을 받지 못하면, **Then** "현재 AI와 연결이 원활하지 않습니다. 잠시 후 다시 시도해주세요." 와 같은 통합된 사용자 친화적 에러 메시지를 표시하고, 재시도 옵션을 제공한다.
- **(예외 케이스 - 시스템 로깅)** **Given** 백엔드가 LLM API 호출 시, **When** API 키 불일치, 시간 초과 등 특정 오류가 발생하면, **Then** 시스템은 오류 유형별로 명확한 로그를 기록하여 신속한 문제 추적을 가능하게 한다.

#### 3. 핵심 구현 원칙 (Key Implementation Principles)

- **INVEST 원칙 기반 질문 생성:** AI가 사용자에게 던지는 질문은 INVEST (Independent, Negotiable, Valuable, Estimable, Small, Testable) 원칙을 기반으로 해야 한다. 이는 사용자의 막연한 아이디어를 점진적으로 실행 가능한 백로그로 구체화하기 위함이며, 백엔드 LLM 프롬프트 설계의 핵심 가이드라인이 된다.

### **Feature 2: AI 기반 스토리 포인트 추정**

#### 1. 사용자 경험 목표 (UX Goal)

- **User's Job (JTBD):** When 주간 계획을 세울 때, **김성장**은 각 과제에 필요한 노력을 객관적으로 가늠하여, **자신이 감당할 수 있는 현실적인 계획을 세우고 싶어한다.**
- **Success Metric:** 이 기능의 성공은 2회차 이상 사용자가 자신의 평균 벨로시티 범위 내에서 주간 계획을 수립하는 비율이 높아지는 것으로 판단할 수 있다.

#### 2. 구현 요건 (Acceptance Criteria)

- **(정상 시나리오 - 포인트 선택)** **Given** 사용자가 백로그에 스토리 포인트를 할당할 때, **When** 포인트 선택 UI를 열면, **Then** 시스템은 피보나치 수열(1, 2, 3, 5, 8, 13)에 따른 선택지를 제공한다.
- **(정상 시나리오 - 분할 제안)** **Given** 사용자가 특정 백로그에 큰 스토리 포인트(예: 8pt 이상)를 할당하려고 할 때, **When** 포인트를 선택하면, **Then** AI는 "8포인트는 다소 큰 작업일 수 있습니다. 'A(3pt)'와 'B(5pt)'로 나누는 것은 어떨까요?" 와 같이 지능적인 작업 분할을 제안하고, 사용자는 이를 수락할 수 있다.
- **(정상 시나리오 - 제안 수락)** **Given** AI가 작업 분할을 제안한 상황에서, **When** 사용자가 [제안 수락] 버튼을 누르면, **Then** 기존 백로그는 유지된 채, 제안된 하위 백로그('A'와 'B')가 새로운 백로그로 즉시 생성된다.
- **(예외 케이스 - 제안 무시)** **Given** AI가 작업 분할을 제안한 상황에서, **When** 사용자가 제안이 부적절하다고 판단하여 [무시하기] 버튼을 누르면, **Then** 제안 UI는 사라지고 사용자의 원래 선택(8pt)이 그대로 유지된다.

### **Feature 3: 스프린트 실행 (Kanban Board)**

#### 1. 사용자 경험 목표 (UX Goal)

- **User's Job (JTBD):** When 한 주를 시작할 때, **김성장**은 이번 주에 집중할 일들을 한눈에 보면서, **진행 상황을 직관적으로 추적하고 관리하고 싶어한다.**
- **Success Metric:** 이 기능의 성공은 스프린트를 시작한 사용자가 최소 1개 이상의 백로그를 '완료' 상태로 이동시키는 비율(스프린트 활성 사용자 비율)이 높은 것으로 판단할 수 있다.

#### 2. 구현 요건 (Acceptance Criteria)

- **(정상 시나리오 - 스프린트 생성)** **Given** 사용자가 백로그 목록에서 백로그를 선택하고 '스프린트 시작하기'를 누르면, **When** 시스템은 **7일의 기간(시작일/종료일 지정 가능)**을 설정하게 하고 사용자가 계획을 확정하면, **Then** 새로운 스프린트가 생성되고 칸반 보드 화면으로 이동한다.
- **(정상 시나리오 - 상태 변경)** **Given** 스프린트가 진행 중인 칸반 보드에서, **When** 사용자가 카드를 드래그 앤 드롭하여 다른 컬럼으로 옮기면, **Then** 카드의 상태가 즉시 서버에 저장되고 화면에 반영된다.
- **(정상 시나리오 - 진행률 업데이트)** **Given** 칸반 보드에서 카드를 '완료(Done)' 컬럼으로 옮기면, **When** 해당 카드의 스토리 포인트가 집계되어, **Then** 스프린트 진행률(예: 8/15pt) 텍스트가 실시간으로 업데이트된다.

### **Feature 4: 스프린트 마감 (Sprint Close)**

#### 1. 사용자 경험 목표 (UX Goal)

- **User's Job (JTBD):** When 한 주간의 스프린트 기간이 끝났을 때, **김성장**은 이번 주에 달성한 성과를 최종적으로 확정하고, **성장과 개선을 위한 다음 단계로 넘어가고 싶어한다.**
- **Success Metric:** 이 기능의 성공은 스프린트를 시작한 사용자가 기간 종료 후 [스프린트 마감하기] 버튼을 눌러 자신의 성과(벨로시티)를 최종 확정하는 비율(스프린트 완료율)이 높은 것으로 판단할 수 있다.

#### 2. 구현 요건 (Acceptance Criteria)

- **(정상 시나리오 - 스프린트 마감)** **Given** 사용자가 진행 중인 스프린트에서, **When** [스프린트 마감하기] 버튼을 누르면, **Then** 시스템은 현재 '완료(Done)' 상태인 백로그들의 총 포인트를 '이번 스프린트의 벨로시티'로 기록하고, 스프린트 상태를 '완료됨'으로 변경하며, 사용자에게 회고를 시작할지 묻는 화면을 보여준다.
- **(정상 시나리오 - 미완료 작업 처리)** **Given** '할 일'이나 '진행 중' 상태의 백로그가 남아있는 상태에서, **When** 사용자가 [스프린트 마감하기]를 누르면, **Then** 해당 미완료 백로그들은 자동으로 다음 스프린트 계획 시 선택할 수 있도록 '백로그 목록'으로 이동된다.
- **(예외 케이스 - 마감 유도)** **Given** 스프린트 종료일이 지났음에도 사용자가 마감하지 않았을 때, **When** 사용자가 서비스에 접속하면, **Then** "지난 스프린트가 아직 진행 중입니다. 지금 마감하고 회고를 진행해볼까요?" 와 같이 마감을 유도하는 모달 또는 배너를 보여준다.

### **Feature 5: 벨로시티 추이 확인 및 계획 지원**

#### 1. 사용자 경험 목표 (UX Goal)

- **User's Job (JTBD):** When 다음 주 계획을 세울 때, **김성장**은 과거 자신의 성과 데이터를 기반으로 **달성 가능한 현실적인 목표를 세우고 싶어한다.**
- **Success Metric:** 이 기능의 성공은 2회차 이상 스프린트를 진행하는 사용자가 자신의 평균 벨로시티 ±20% 범위 내에서 다음 스프린트 목표를 설정하는 비율이 높아지는 것으로 판단할 수 있다.

#### 2. 구현 요건 (Acceptance Criteria)

- **(정상 시나리오 - 벨로시티 계산)** **Given** 사용자가 1회 이상 스프린트를 '마감'했을 때, **When** 시스템은 **완료된 최근 3개 스프린트**의 확정된 벨로시티 평균값을 계산하여, **Then** 이 값을 해당 사용자의 '평균 벨로시티'로 관리한다.
- **(정상 시나리오 - 계획 지원)** **Given** 사용자가 새로운 스프린트를 계획하기 위해 백로그를 선택할 때, **When** 화면에는 사용자의 평균 벨로시티 정보(예: "나의 평균 벨로시티: 13pt")가 표시되어, **Then** 사용자는 이를 참고하여 현실적인 목표를 설정할 수 있다.
- **(정상 시나리오 - 목표 초과 경고)** **Given** 사용자가 자신의 평균 벨로시티를 초과하여 스프린트 목표를 설정하려 할 때(백로그를 선택할 때), **When** 선택한 포인트의 합계가 평균 벨로시티를 넘어서면, **Then** 시스템은 "평균 벨로시티를 초과합니다. 계획을 조정하는 것이 어떨까요?" 와 같이 경고성 텍스트를 표시하지만, 사용자는 이를 무시하고 계획을 확정할 수 있다.

### **Feature 6: AI 기반 회고 (AI Retrospective)**

#### 1. 사용자 경험 목표 (UX Goal)

- **User's Job (JTBD):** When 한 주간의 스프린트를 마쳤을 때, **김성장**은 자신의 실행 결과를 객관적으로 돌아보고, **다음 주에 더 나은 성과를 낼 수 있는 구체적인 방법을 찾고 싶어한다.**
- **Success Metric:** 이 기능의 성공은 스프린트를 마감한 사용자가 최종적으로 'AI 회고 요약 리포트'까지 생성하는 비율(회고 전환율)이 높게 유지되는 것으로 판단할 수 있다.

#### 2. 구현 요건 (Acceptance Criteria)

- **(정상 시나리오 - 데이터 기반 질문)** **Given** 사용자가 '회고 시작하기'를 누르면, **When** AI는 해당 스프린트의 데이터(계획 대비 달성률, 완료/미완료 백로그 등)를 기반으로 "이번 주 목표는 15pt였는데, 13pt를 완료하셨네요. 특히 '새로운 기능 리서치' 백로그를 마무리하지 못했는데, 어떤 어려움이 있었나요?"와 같이 개인화된 첫 질문을 던진다.
- **(정상 시나리오 - 대화 및 요약)** **Given** 사용자가 AI의 질문에 답변하며 대화를 이어가면, **When** 사용자가 [요약하고 마치기] 버튼을 누르면, **Then** AI는 전체 대화 내용을 바탕으로 "Keep(유지할 점)", "Problem(개선할 점)", "Try(시도할 점)" 형식의 KPT 요약 리포트를 생성하여 보여준다.
- **(예외 케이스 - 조기 종료)** **Given** 사용자가 회고 대화 중, **When** 더 이상 대화를 이어가고 싶지 않다면, **Then** 언제든지 [요약하고 마치기] 버튼을 눌러 현재까지의 대화 내용만으로 리포트를 생성하고 회고를 마칠 수 있다.

## 2. 최종 기술 결정사항

- **인증 방식:** JWT (JSON Web Token) 기반 이메일 / 비밀번호 인증
  - **Access Token 만료 시간: 7일** (Refresh Token은 MVP 범위에서 제외)
- **프론트엔드 스택: Svelte5 + Tailwind CSS + shadcn-svelte**
- **API 에러 응답 표준:** **RFC 7807 Problem Details 사용**
- **스토리 포인트 단위:** **피보나치 수열 (1, 2, 3, 5, 8, 13) 사용**
- **데이터베이스:** AWS RDS for PostgreSQL
- **LLM 모델:** OpenAI GPT-5
- **클라우드 제공자:** Amazon Web Services (AWS)
- **이메일 발송:** Amazon SES (Simple Email Service)
